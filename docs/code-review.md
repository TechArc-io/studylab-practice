# Code review

A useful pull request lets another engineer understand the problem, solution, risk, and verification without reconstructing your thinking.

## Author checklist

- State the user-facing or engineering problem.
- Explain the chosen approach and important tradeoffs.
- List the exact commands you ran and their results.
- Include keyboard, screen-reader, responsive, and error-state notes when relevant.
- Identify uncertainty honestly; do not claim checks you did not run.

## Responding to feedback

Read the entire review before editing. Group related comments, ask when the expected behavior is unclear, and reply with evidence. A review comment is not complete because the code changed; it is complete when the concern is addressed and the checks still pass.
